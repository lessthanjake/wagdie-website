The Button component is used for triggering actions or navigation. It supports multiple visual variants and states for different use cases.

## Features

- Three visual variants: primary, secondary, and danger
- Three sizes: small, medium, and large
- Disabled state support
- TypeScript support with full type safety
- Tailwind CSS styling
- Accessible by default

## Usage

Use buttons for primary actions in your interface. Choose the appropriate variant based on the action's importance:

- **Primary**: Use for the main action in a dialog, form, or page
- **Secondary**: Use for less important actions that complement the primary action
- **Danger**: Use for destructive actions like delete, remove, or confirm removal

### Different Sizes

You can adjust the button size based on the context:

- **Small**: Use in tight spaces like tables or compact layouts
- **Medium**: Default size for most use cases
- **Large**: Use for important call-to-action buttons

### States

Buttons support different states:

- **Enabled**: Default state with full interactivity
- **Disabled**: Shows but cannot be clicked, useful for showing unavailable actions

## Accessibility

- Buttons have proper semantic `<button>` element
- Disabled state prevents keyboard interaction
- Screen readers can identify button state and purpose

## Implementation Details

The Button component is built with:
- **React 18** for the component logic
- **TypeScript** for type safety
- **Tailwind CSS** for styling
- No external dependencies beyond React

## Design Tokens

The component uses consistent design tokens:

- **Border radius**: `rounded-md` (0.375rem)
- **Font weight**: `font-medium` (500)
- **Transitions**: `transition-colors` for smooth color changes
- **Focus ring**: `focus:ring-2` with appropriate colors per variant

## Best Practices

1. **Use clear, actionable labels**: Button text should describe what happens when clicked
2. **Choose the right variant**: Match the button's visual weight to its importance
3. **Maintain consistency**: Use the same button styles for similar actions across your app
4. **Consider hierarchy**: Primary actions should be visually distinct from secondary actions
5. **Provide feedback**: Use loading states or disabled states when actions are unavailable

## Common Patterns

### In Forms

```tsx
<form onSubmit={handleSubmit}>
  <input type="email" placeholder="Email" />
  <Button type="submit" variant="primary">
    Sign Up
  </Button>
  <Button type="button" variant="secondary">
    Cancel
  </Button>
</form>
```

### In Dialogs

```tsx
<Modal isOpen={showModal} title="Confirm Action">
  <p>Are you sure you want to delete this item?</p>
  <div className="flex gap-2 justify-end">
    <Button variant="secondary" onClick={handleCancel}>
      Cancel
    </Button>
    <Button variant="danger" onClick={handleConfirm}>
      Delete
    </Button>
  </div>
</Modal>
```

### With Icons

```tsx
<Button variant="primary">
  <PlusIcon className="w-4 h-4 mr-2" />
  Add Item
</Button>
```
